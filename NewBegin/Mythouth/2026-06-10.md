## [Medium] 零钱兑换 (322)

### 🤔 我的原始思路
- 每个 amount 尝试减去 coins[i]
- 贪心：sort(coins)，先取最大的，大于 amount 就换小的
- 初始化为 INT_MAX/2
- dp[amount] == INT_MAX/2 return -1; else return dp[amount]

### 🔍 我的思考过程
1. **贪心不可行**：coins=[1,3,4], amount=6 时贪心 4+1+1=3 不如 3+3=2
2. **回到 279 的框架**：dp[i] = min(dp[i - coin] + 1) over all coins
3. **初始化问题**：coins 可能不含 1，用 INT_MAX/2 作为不可达标记，避免溢出
4. **跳过条件**：dp[i - coin] 不可达时跳过 min 比较
5. **循环边界**：踩坑 → `i <= amount` 而不是 `<`，否则 dp[amount] 永不计算

### 💡 对比收获
- 322 和 279 是同一类最值 DP，面额来源不同（平方数 vs 数组）
- 贪心在通用面额下可能失败，DP 保证最优
- 不可达标记 + 跳过条件是最值型 DP 处理"无解"的标准技巧
- 统一框架：70（面额1/2）、279（平方数）、322（数组）→ 同一模板

## [Medium] 最长递增子序列 (300)

### 🤔 我的原始思路
- dp[i] 表示以 nums[i] 结尾的 LIS 长度
- "转移方程一维只需要一个，不需要所有"（初稿，后来修正）
- 维护全局最大值 ans

### 🔍 我的思考过程
1. **状态定义**：dp[i] = 以 nums[i] 结尾的最长递增子序列长度
2. **最初的误区**：以为转移只需要看前一个元素，实际需要遍历所有 j < i
3. **修正**：dp[i] = max over j < i (dp[j] + 1) 当 nums[j] < nums[i] — 和 322 遍历硬币本质一致
4. **全局最大值**：答案是所有 dp[i] 的 max，不是 dp[n-1]（LIS 不一定以最后一个元素结尾）
5. **边界细节**：ans 初始化为 1 而不是 0（n=1 时不会进循环，但答案应为 1）

### 💡 对比收获
- **新的状态定义模式**："以 i 结尾" vs "凑成目标 i" — 两种不同的 DP 视角
- **遍历对象不同**：322 往前减 coin，300 往前遍历 j — 但都是"外层目标 + 内层候选"的结构
- **dp 初始化为 1**：每个元素自身至少构成长度 1 的子序列
- **ans 初始化**：n≥1 时至少返回 1，n=0 不会出现（题目约束）
- **O(n log n) 优化**：贪心 + 二分（tails 数组）替代 O(n²)
  - lower_bound 找第一个 ≥ num 的位置（就是 35 题 Search Insert Position）
  - 替换成更小的值，贪心让后面的数更容易接上
  - tails 不是 LIS 本身，只保证长度正确
